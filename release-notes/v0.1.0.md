**First crates.io release.** holt is an embedded adaptive radix
tree for path-shaped metadata — file paths, S3 object names,
multi-tenant namespaces — built for sub-microsecond lookups,
per-blob concurrency, and crash-safe persistence.

> ⚠️ **Pre-1.0.** Pin the exact version (`holt = "=0.1.0"`).
> Minor releases may break source compatibility until `1.0`.

## Highlights

- **Adaptive radix tree core** — 9 `NodeType` variants, bit-packed
  slot table, 4 KB blob header, compile-time-asserted byte
  offsets, SIMD-accelerated `Node16` search (SSE2 / NEON / scalar).
- **Multi-blob via in-tree crossings** — once a 512 KB blob fills,
  `splitBlob` migrates a subtree into a fresh blob and installs
  a `BlobNode` placeholder. `compactBlob` repacks in place,
  `mergeBlob` folds shrunk children back into the parent. Trees
  grow to arbitrary size in 512 KB increments.
- **Per-blob 3-mode `HybridLatch`** — optimistic / shared /
  exclusive, after LeanStore. Readers fast-path is wait-free; no
  Tree-wide writer mutex.
- **Physiological WAL** — 11 `TxnOp` variants (incl. `Batch` for
  user-level multi-op atomicity), CRC32-protected records,
  64 KB group-commit auto-flush, torn-tail-tolerant replay.
- **Stateful range iterator** — `Tree::range().prefix(p)
  .start_after(k).delimiter(b'/')` with S3-style `CommonPrefix`
  rollup, anchored descent so prefix queries don't scan the full
  tree.
- **Atomic batched transactions** — `Tree::txn(|batch| { ... })`
  collapses N ops into one WAL record; either all replayed or
  none.
- **Persistent backend** — single packed `blobs.dat` + atomic-
  rename manifest, `O_DIRECT` on Linux, `F_NOCACHE` on macOS.

## How does it compare?

On the **path-shaped metadata workloads holt is built for**,
this release beats RocksDB and SQLite on every measured axis
(Apple M-series, `N = 20 000`):

| op | holt | rocksdb | sqlite |
|---|---|---|---|
| `objstore_get` (S3-style path) | **190 ns** | 554 ns | 534 ns |
| `fs_get` (POSIX path) | **196 ns** | 625 ns | 538 ns |
| `objstore_list` (prefix scan, 100 entries) | **10.7 µs** | 17.7 µs | 13.2 µs |
| `fs_list` (prefix scan, 100 entries) | **10.7 µs** | 18.9 µs | 13.4 µs |

On **random 32-byte KV** (the ART anti-pattern, included on
purpose as a fairness check) holt loses badly — ~30× slower than
both alternatives. Pick the engine that matches your **key shape**.

Methodology + reproduction in [`benches/README.md`](benches/README.md).

## Compatibility

- **Rust 1.82+** (MSRV pinned).
- **Unix only** — Linux + macOS. The crate `compile_error!`s on
  Windows targets by design.
- **MIT licensed.**

## Install

```toml
[dependencies]
holt = "=0.1.0"
```

API: <https://docs.rs/holt/0.1.0>

## What's queued for v0.2

Performance + concurrency upgrades, not new API surface:
`io_uring` backend on Linux, SIMD CRC32, background checkpoint
thread, per-node `HybridLatch` for finer-grained concurrent reads,
range-iterator fast-forward in delimiter mode. See
[`ROADMAP.md`](ROADMAP.md) for the full list.

## Scope reminder

holt is **a single-node embedded metadata engine**, not an S3
server or a distributed coordinator. Items explicitly out of
scope: Windows, replication / consensus, network frontend,
multi-tenant bucket registry, SQL, vector search, full-text
search. See `ROADMAP.md`'s "Not on the roadmap" section.
